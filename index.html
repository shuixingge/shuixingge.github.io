Android UI 卡顿及ANR检测原理

发表于 2017-04-13

一：背景

众所周知，Android不允许在UI线程中做耗时的操作，否则有可能发生ANR的可能，默认情况下，在Android中Activity的最长执行时间是5秒，BroadcastReceiver的最长执行时间则是10秒。如果操作这个时长，则会产生ANR。而本文所要介绍的主要内容是检测UI线程中的耗时操作，从而能够定位一些老代码中的各种耗时的操作，作为性能优化的依据。

private final Runnable _ticker = new Runnable() {
@Override public void run() {
_tick = (_tick + 1) % Integer.MAX_VALUE;
}
};

@Override
public void run() {
setName(“|ANR-WatchDog|”);

int lastTick;
int lastIgnored = -1;
while (!isInterrupted()) {
    lastTick = _tick;
    _uiHandler.post(_ticker);
    try {
          //睡眠一段时间，确保_tick的值更新后再做判断。
        Thread.sleep(_timeoutInterval);
    }
    catch (InterruptedException e) {
        _interruptionListener.onInterrupted(e);
        return ;
    }

    // If the main thread has not handled _ticker, it is blocked. ANR.
    if (_tick == lastTick) {
        if (!_ignoreDebugger && Debug.isDebuggerConnected()) {
            if (_tick != lastIgnored)
                Log.w("ANRWatchdog", "An ANR was detected but ignored because the debugger is connected (you can prevent this with setIgnoreDebugger(true))");
            lastIgnored = _tick;
            continue ;
        }

        ANRError error;
        if (_namePrefix != null)
            error = ANRError.New(_namePrefix, _logThreadsWithoutStackTrace);
        else
            error = ANRError.NewMainOnly();
        _anrListener.onAppNotResponding(error);
        return;
    }
}

OkHttp源码分析(一)

发表于 2017-02-07

前言

OkHttp简介： OkHttp是一个网络通信框架，支持HTTP和HTTP/2的客户端。在OkHttp出现，Android平台上的Http库一般使用HttpClient或者HttpUrlConnection，这两者各自的特点和缺点，想必大家都已经都十分的清楚。具体表现为：

OkHttp特点与优势：

  • 支持同步和异步请求。同步请求会阻塞当前的线程
  • 支持流式传输，所以支持大文件的下载
  • 支持Http 2.0和Http 1.x，对于Http 2.0，同一个服务器的所有请求共用一个Socket连接，而Http 1.x通过连接池复用来实现连接复用。
  • 当服务器配置了多个IP地址，当第一个IP连接失败的时候，OkHttp会自动尝试下一个IP。
  • 支持gzip压缩
  • 通过interceptor来实现对请求的各种处理

同步和异步请求

同步请求

同步调用即调用realCall.execute()方法:

  • 首先会判断这个call是否执行过，每个请求只允许执行一次，否则就会抛出异常;

  • 然后通过Disptacher类来执行这个call;

  • 最后通过getResponseWithInterceptorChain()来获取响应.

Dispatcher 顾名思义为一个分发器，同步执行call时， dispatcher().executed(this)的逻辑为：把这个call 加入到队列runningSyncCalls中。

异步请求

异步则是执行RealCall.enqueue()方法，同理一个异步请求也只允许执行一次，然后直接调用Dispatcher的enqueue(),

equeue()的时候首先判断异步队列runningAsyncCalls里的请求是否小于最大的请求数或者每个主机的请求数小于阈值。如果小于则把当前的call加入到runningAsyncCalls中，并把这个这个call加入到线程池中，否则加入到等待队列readyAsyncCalls中。

我们再来看一下AsyncCall，这个类继承了NamedRunnable,而NamedRunnable是一个Runnable,在NamedRunnable的run()方法中调用了AsyncCallexecute()方法，并且通过getResponseWithInterceptor()方法获取了相应。

基本用法

  • 首先构建retrofit对象， 这里采用的Builder模式来构建复杂对象

retrofit的基本用法就是这样，在上面一个指的注意的一点: GitHubService只是一个抽象的接口，那么怎么能够直接调用它的抽象方法呢？

CallAdapter

前面也提到了CallAdapter，CallAdapter.adapt()方法将请求结果的返回成特定的类型然后返回，当不指定具体的CallAdapter的会使用特定的DefaultCallAdapter,除此之外还有支持返回Oberserable的RxJavaCallAdapter. 我们看一下CallAdapter的注释：把Call转化为类型 T,这个T就是范型类型，通过CallAdapter.Factory创建CallAdapter实例，并且通过Retrofit.Builder的addaddCallAdapterFactory(Factory) 添加到Retrofit实例中。

看一下具体的DefaultCallAdapterFactory,可以看到直接把OkHttpCall返回了。

我们可以自己定义一个CallAdapter，根据CallAdapter接口的定义形式, CallAdapter 使用了工厂来创建，CallAdapter.Factory主要有三个方法，get返回创建的CallAdapter的实例，getParameterUpperBound返回泛型的上界，getRawType返回原始类型。

我们在来具体看一下具体的RxJavaCallAdapter，由于这部分代码比较多，所以我们分部分贴出来。 首先来看一下get()方法

  • RxJavaAdapterFactory.get():主要做了一些类型判断，判断getRawType()返回的是不是Observable类型，并通过getCallAdapter来获取CallAdapter的实例。
  • getCallAdapter（）：返回具体的CallAdapter类型，这里使用了策略模式，主要的逻辑是根据 getParameterUpperBound返回的泛型的上界的原始类型，来确定使用哪种CallAdapter，默认的使用SimpleCallAdapter.

  • SimpleCallAdapter:可以看到返回了Observable 类型，CallOnSubscribe<>实现了OnSubscribe<>，并在OnSubscribe<>的next方法进行了网络请求

  • CallOnSubscribe:执行网络请求。
